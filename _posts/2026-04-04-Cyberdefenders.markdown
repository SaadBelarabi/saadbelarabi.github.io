Hello, this is a write-up for a digital forensics lab called "QRadar101". For those who don't know what this mean, we have a simulated environment with different resources (mentioned after the scenario), and a bunch of questions, that are supposed to sort of lead us to uncover what happened, if there's something more to what's mentioned in the scenario and try to recreate the timeline of events. It doesn't matter if you've never tried this kind of stuff yourself or ever heard of it. I tried to answer each question in a way that would be understandable for whoever has at least a basic knowledge in IT. Hope you have a good read and we'll meet at the end if this write-up!

## Scenario *(taken from the lab page itself)*:

> A financial company was compromised, and they are looking for a security analyst to help them investigate the incident. The company suspects that an insider helped the attacker get into the network, but they have no evidence.
>
> The initial analysis performed by the company's team showed that many systems were compromised. Also, alerts indicate the use of well known malicious tools in the network. As a SOC analyst, you are assigned to investigate the incident using QRadar SIEM and reconstruct the events carried out by the attacker.

## Q2: What is the IDS software used to monitor the network?

From the list of log sources we found previously, only one is an IDS (Intrusion Detection System): suricata. An IDS is a rule-based engine that detects patterns (could be a combination of one/multiple of: IPs, ports, flow, payload). Whenever these patterns are matched, alerts are generated for the analyst so he can investigate and see whether this was an actual malicious behavior or just a false positive.

**Answer: `Suricata`**

## Q3: What is the domain name used in the network?

Since we are in an AD environment, we can look at logs that mention logon attempts. These will most of the time generated by the Domain Controller, which handles authentication, as well as other tasks.

Windows Event Log normally logs these kind of events, under the logs with Event ID 4624 and 4625, so we can use this as a filter and take whatever log we want. The payload should normally contain the computer account attempting to logon and the domain they are part of.

```
Account Name: HD-FIN-02$    Account Domain: HACKDEFEND
```

**Answer: `HACKDEFEND` (still needs rectification because the answer is HACKDEFEND.local)**

The scenario mentions that potentially, an insider helped the attacker get into the network. We can then look for logs that have as a destination IP an internal IP, likely 192.168.x.x. We'll look into that an see what comes up.

## Q10: Hackers do not like logging, what logging was the attacker checking to see if enabled?

A common info that attackers look for when performing their attack is checking whether commands (in powershell or cmd) are being logged. The following are 2 keys from registry that show this info (I believe by default these aren't unless activated by and admin or whoever has the perms to do so):

```
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System\Audit
HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging
```

Looking at events generated with Event IDs 4656, 4663 or 4658 (or 13,14 and 15 for a more noise-free result) hasn't been very fruitful, maybe because Audit Object Access policy isn't set, or because no registry key was actually accessed. Another attempt looking for payloads containing those registry keys was successful though.

Filter:

![img](/images/qradar101/image014.png)


Result:

![img](/images/qradar101/image015.png)


So the attacker was actually checking if powershell commands were logged.

**Answer: `powershell`**

## Q11: Name of the second system the attacker targeted to cover up the employee?

My thought process is the following: suppose a machine is targeted to cover up the suspect employee, some actions were performed on that machine to incriminate it and direct the intention to that machine. Now this machine either is accessed directly or through lateral movement from another infected machine. So far HD-FIN-03 is the only machine used by a worker we know is infected. So whatever we are looking for, it's not in that machine, nor in the DC.

Unfortunatly I couldn't have reached a promising result without looking at quite an amount of logs. What I found (through the IDS alerts) is an exfiltration attempt of a file named sami.xlsx from the IP 192.168.10.29 that corresponds to the HD-FIN-02 machine.

![img](/images/qradar101/image016.png)


The attacker after that removes the file.

Later on after more investigation on this file, we see that the MGNT-01 machine tried to remove the same file, first with the "rm" command then with the "del" command.

```
cmd.exe /Q /c rm sami.xlsx 1> \\127.0.0.1\ADMIN$\__1604917981.0572538 2>&1
cmd.exe /Q /c del sami.xlsx 1> \\127.0.0.1\ADMIN$\__1604917981.0572538 2>&1
```

I don't know if this is how the cover up was intended (and also to make the targeted person look dumb) it seemed to me kind of pathetic, so I wasn't really convinced, even though it ended up being the right answer.

**Answer: `MGNT-01`**

## Q12: When was the first malicious connection to the domain controller (log start time - hh:mm:ss)?

A first connection means an authentication attempt by the attacker, whatever the vector is from. Since we know HD-FIN-03 is already compromised, it's not a bad guess to think we might log onto DC from that machine. Of course there's the possibility (I THINK) that the connection might be possible from another machine, but we'll check first if there was an attempt to do so from a remote machine.

Filter:

Results:

It's worth noting that before the connection, there was a very likely host discovery attempt from the same machine, on 192.168.10.0/24 and 192.168.20.0/24 on port 8, which is unusual partially because source ports have high values when whatever action is coming from them.

![img](/images/qradar101/image019.png)

## Q14: What is the MITRE persistence technique ID used by the attacker?

In Q10 we searched up for indicators for registry access, and even though the results weren't interesting at that time, some stuff came up worth keeping in mind. We found:

- A set/change of the value of the ImagePath of a startup program by a vbs (visual basic) script (T1547.001)
- A named pipe listening `\\[.]\pipe\oxcvuvf` on the service "oxcvuvf" (likely an attempt of token theft/T1134.001)
